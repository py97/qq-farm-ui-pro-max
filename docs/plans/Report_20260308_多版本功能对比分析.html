二、🚨 最关键差异：WASM 加密协议

⚠️ 阻断级别警告 三个参考版本已完成 WASM 加密协议适配，本项目尚未集成，将导致无法与腾讯服务器正常通信。这是当前最高优先级的适配任务。

2.1 新加密协议核心原理

  • 加密方式：从 .tqapkg 小游戏分包提取 tsdk.wasm（119KB），含 C/C++ 编译的高强度加密算法
  • 非对称模式：请求体必须加密，响应体保持明文 Protobuf

2.2 集成改动要点

参考 P大版本 network.js，核心改动为 encodeMsg 函数：

- function encodeMsg(serviceName, methodName, bodyBytes) {
-     const msg = types.GateMessage.create({
-         body: bodyBytes || Buffer.alloc(0), ...
-     });
+ const cryptoWasm = require('./crypto-wasm');
+
+ async function encodeMsg(serviceName, methodName, bodyBytes) {
+     let finalBody = bodyBytes || Buffer.alloc(0);
+     if (finalBody.length > 0) {
+         finalBody = await cryptoWasm.encryptBuffer(finalBody);
+     }
+     const msg = types.GateMessage.create({
+         body: finalBody, ...
+     });
⚠️ 重要提示 响应体不需要解密！新协议逆向记录明确指出，对明文数据调用 decryptBuffer 反而会破坏数据导致 Protobuf 解码异常。

四、网络层核心差异详解

4.1 消息编码流程对比

P大版本（已适配新协议）：

bodyBytes → cryptoWasm.encryptBuffer(bodyBytes) → GateMessage.encode → ws.send

本项目（旧协议，待适配）：

bodyBytes → GateMessage.encode → ws.send  （无加密步骤）

4.2 消息解码流程

📌 四个版本一致 服务端响应体为明文 Protobuf，直接 GateMessage.decode(buf)。新版协议是「非对称」加密——客户端请求须经 WASM 加密处理，接收的状态数据仍为明文。

4.3 本项目独有网络高级机制

  1. 令牌桶全局限流：334ms 间隔 + ±20% 随机抖动 → 防止匀速行为指纹
  2. 双队列优先级：urgent/normal + Anti-Starvation 防饿死（每 10 个紧急释放 1 个普通）
  3. 断路器模式：1002003 风控 → 全局熔断 → 队列清空 → 半开恢复
  4. 请求 TTL 丢弃：好友列表 3s / 常规 5s / 核心操作 15s
  5. 防雪崩重连抖动：5~25 秒随机延迟，分散重连峰值
