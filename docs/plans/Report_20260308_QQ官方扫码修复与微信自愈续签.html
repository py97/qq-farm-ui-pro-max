修复内容

1. QQ 官方扫码恢复

  • core/src/services/qrlogin.js 恢复使用官方 GetLoginCode、syncScanSateGetTicket、ide/login 三段式流程。

  • 当 wx_car / wx_ipad 启动后收到 ws_error 400，系统会自动使用已保存的 wxid 调用 JSLogin 换取新 code，然后自动重启该账号。
  • 真实日志已确认账号 30 在一次失败后自动换出新 code 并重新登录成功。

3. 账号持久化增强

  • core/src/models/store.js 现在将 uin / qq / code 同步写入 auth_data，并在重载时优先恢复正确字段。
  • 该修复针对本轮实测中暴露出的核心缺陷：微信账号重启后数据库中只剩错误的数值 uin，导致后续握手参数失真。

当前剩余问题

唯一剩余的业务阻塞： QQ 账号 1005 当前数据库里保存的还是旧的失效/临时登录码。

这不是新的代码缺陷，而是旧账号数据还没有完成一次基于新流程的重新授权。现在系统已经能正常生成官方 QQ 二维码，也能正常轮询状态；只差你在 3000 面板里对该账号做一次“扫码更新”。

  • 建议操作：打开账号弹窗，选择 QQ，直接扫码，不要手动粘贴 qzone/qqq/code/<32位码> 这种临时登录链接。
  • 当前前端已对这种 URL 增加拦截提示，避免再次把临时二维码链接误当成最终授权码。

优化建议

  • 给 QQ 账号也补“失效态可视化”： 当前 1005 已停止但 API 快照里没有显式保留 wsError，建议把 expired_auth 作为稳定状态字段写回账号表，避免 UI 上看起来只是“停止”。
  • 把微信辅助字段单独持久化： 现在 qq 辅助字段仍可能在部分更新链路里被清空，建议把微信账号的数值 QQ 号也固定写入 auth_data.qq 并在 API 输出时显式保留。
  • 将自动续签逻辑下沉成统一服务： 目前微信续签逻辑在 worker-manager 内部，后续可以抽成独立的“Auth Refresh Service”，统一处理 wx_car / wx_ipad / wx。
  • QQ 手动输入模式增加更强校验： 建议在服务端也识别明显的临时二维码 URL/码格式，直接拒绝落库，避免再次留下会立即 400 的坏账号。
