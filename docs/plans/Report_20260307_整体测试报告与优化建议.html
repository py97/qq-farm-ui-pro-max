关键问题与证据

P1-01

普通用户可直接进入管理员页面外壳，前端角色守卫缺失

普通用户 admin1 登录后，侧边栏中的管理员菜单项已经隐藏，但手动访问 /users 时仍能打开“用户管理”页面骨架，页面随后因数据接口 403 显示“请求失败”。这说明当前前端只做了菜单隐藏，没有做路由级角色阻断。

实测证据：普通用户直接打开 /users，页面标题显示“用户管理”，表格为空，toast 提示 请求失败: Request failed with status code 403。
代码定位：/Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/web/src/router/index.ts:87-115 只验证是否登录；/Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/web/src/components/Sidebar.vue:320-328 仅隐藏菜单；管理员路由只在 /Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/web/src/router/menu.ts:74-105 标注 adminOnly，但未参与路由前置守卫判断。
P1-02

运行中的账号实际未连接，挂机主链路当前不可用

管理员账号视角下，系统显示有两个账号处于运行中，但两者都返回 connected: false，并在仪表盘日志中持续报出 [WS] 错误: Unexpected server response: 400 与“连接被拒绝，可能需要更新 Code”。这不是单纯 UI 问题，而是实际业务运行态故障。

实测证据：/api/accounts 返回账号 30 与 1004 均为 running: true 且 connected: false；仪表盘日志连续出现 WebSocket 400。
影响：成熟秒收、好友巡查、流程编排、通知触发等核心链路当前无法被视为可用。
P1-03

账号数据存在空用户名记录，租户归属存在完整性风险

管理员接口返回的账号列表中存在一条 id = 1004 的记录，其 username 为空字符串，但账号依然处于运行态。这会让“账号必须归属于某个用户”的假设失效，后续权限过滤与数据审计都可能出现灰区。

实测证据：/api/accounts 返回 { id: 1004, username: "", running: true, connected: false }。
代码定位：/Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/core/src/controllers/admin.js:1104-1112 对普通用户只按 a.username === req.currentUser.username 过滤。这意味着 ownerless 账号会被系统视为特殊状态，必须在持久化层面明确禁止或单独标识。
P2-01

下线删除秒数默认值与真实返回值不一致

前端设置仓库把 offlineDeleteSec 默认值声明为 120，但当前两个账号真实的 /api/settings 返回都为 0，设置页实际显示也是 0。如果 0 被解释为立即删除、关闭删除，或者保留默认，都可能导致运营理解与系统行为不一致。

实测证据：账号 30 与 1004 的 /api/settings 均返回 offlineReminder.offlineDeleteSec = 0。
代码定位：前端默认值为 /Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/web/src/stores/setting.ts:130-138；后端返回路径在 /Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/core/src/controllers/admin.js:1091-1097。
P2-02

普通用户保存全局主题时接口返回成功，但实际不生效

普通用户调用 /api/settings/theme 时，接口返回 200 OK，但全局主题配置保持不变。这会把“无权限修改”伪装成“保存成功”，属于典型的静默成功问题。对最终用户来说，这是设置成功假象，不利于排查。

实测证据：普通用户提交主题切换请求后，接口返回 200 且数据仍为原值 { theme: "dark" }，随后再次读取 /api/ui-config 结果未变化。
代码定位：/Users/smdk000/文稿/qq/qq-farm-bot-ui-main_副本/core/src/controllers/admin.js:985-1004 对非管理员直接返回当前 UI 配置，不写库。
P3-01

构建对外网字体与若干 UnoCSS 图标有隐式依赖

pnpm build:web 虽然最终成功，但在受限网络下会报 Google Fonts 拉取失败，并且多组 UnoCSS 图标名无法加载。这会降低离线构建的稳定性，也会让产物质量依赖外部网络。

实测证据：构建输出出现 Failed to fetch web fonts: https://fonts.googleapis.com/...、getaddrinfo ENOTFOUND fonts.googleapis.com，同时反复出现 failed to load icon。
影响：部署环境若断外网、DNS 收敛或被墙，构建日志会持续报错，视觉一致性与可复现性下降。

优化建议

1. 给前端路由补齐角色守卫，禁止普通用户挂载管理员视图

建议把 menuRoutes 中的 adminOnly 纳入 router.beforeEach 判定。在拿到当前登录用户角色后，遇到管理员页面应直接重定向到首页或 403 页面，而不是先渲染再由接口报错。

2. 先处理 WebSocket 400 与“running 但 disconnected”矛盾状态

这是当前最直接影响可用性的主故障。需要检查 Code 版本、握手协议、服务端 WS 路径、账号登录平台参数与自动重连逻辑。建议同时把前端状态拆成“已启动 / 已连接 / 最近错误”三层，避免把不可用账号继续呈现为正常运行。

3. 对 ownerless 账号做数据迁移和持久化约束

禁止新增或保存 username = "" 的账号。若系统必须支持“公共账号”，应显式增加 ownerType = system 之类的字段，而不是用空字符串混用用户归属语义。

4. 统一下线删除秒数的默认语义与校验策略

明确 0 的业务含义，然后同步修复前端默认值、后端默认返回值、页面说明文案与保存校验。当前至少需要让“默认值”和“实际回显值”保持一致。

5. 取消普通用户的“静默成功”保存体验

对于普通用户修改全局主题、下线提醒等无权限操作，建议返回明确的权限错误，或者由前端在提交前直接隐藏保存按钮并提示“仅管理员可修改全局配置”。这比返回 200 但不生效更易维护。

6. 为全局设置类接口增加一致的 Schema 校验

建议把 theme、offlineReminder、trialConfig、thirdPartyApi、clusterConfig、timingConfig 纳入统一的类型、枚举、边界与空值校验体系，避免配置入口分散后出现行为漂移。

7. 清理外部构建依赖，保证离线可构建

建议把 Google Fonts 改为本地打包字体或系统字体降级，并核对 UnoCSS 图标名是否写错。当前构建虽然成功，但部署环境对外网依赖仍然偏强。

8. 建立最小自动化回归集

优先补 8 组自动化：登录锁定、普通用户 API 403、普通用户路由守卫、体验卡冷却、卡密复用失败、主题保存权限、/api/accounts 租户隔离、设置回显一致性。这样能最快覆盖这次实测发现的核心风险。
