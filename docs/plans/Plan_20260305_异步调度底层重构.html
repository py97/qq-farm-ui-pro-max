异步调度底层重构与健壮性改造计划

📅 日期: 2026-03-05 👤 作者: Antigravity 🎯 目标模块: core/src/services/(farm.js | friend.js) 高优先级架构重构

📌 1. 业务背景与问题定义 (Problem Statement)

在 `v4.2.0` 的演进中，我们成功引入了 common.js。该模块包含企业系统级别的异步包装函数：

  • withTimeout (防止网络请求永久挂死)
  • withRetry (指数退避熔断重试机制)
  • createDailyCooldown (CD冷却时间节流阀)
  • createDailyTaskManager (防并发重入的任务管理锁)

痛点所在： 目前 `farm.js` 和 `friend.js` 中仍然充斥着大量原始、脆弱的 await sleep(100) 以及缺乏 try-catch-timeout 保护的裸奔 sendMsgAsync 调用。如果腾讯服务器发生丢包或微服务假死，整个机器人的该用户上下文（Context）可能发生卡死阻塞且不再恢复，这是极其危险的技术债。

🏗️ 2. 架构设计与实现策略 (Architecture Design)

我们的核心思想是 "消费/生产分离，优雅降级"。将原始的网络发送层通过包装器（Decorators）包裹，并在业务侧捕获并进行静默丢弃（优雅降级），确保巡检循环永不崩溃。

⚠️ 4. 影响面与风险分析 (Impact Analysis)

4.1 会破坏什么？(Risks)

  • 死锁隐患: `withRetry` 默认如果不设置抛弃错误，可能导致某个请求在 3 次退避中消耗长达 10~20 秒，从而拖慢其他并发请求。必须要仔细设定 `maxRetries: 2` 以及 `delay: 500ms`。
  • 防偷时序冲突: 农场所设计的「防偷 60s 抢收 `harvestUrgent`」是一个**对延时极度敏感**的操作。它不允许被套上 `withRetry` 长时间悬挂，必须一击即退！

4.2 解决对策 (Mitigation)

// 伪代码：在 farm.js 中，针对常规包使用保守防御，针对抢收包使用裸奔/或仅包裹 Timeout
// 常规地块获取 (允许重试)
const lands = await withRetry(
    () => withTimeout(() => sendMsgAsync(req), 5000), 
    3, 500
);

// 极限防偷抢收 (仅设置极限超时防御，不重试，失败拉倒)
const success = await withTimeout(
    () => sendMsgAsyncUrgent(harvestReq), 
    3000
).catch(e => {
    logger.warn(`防偷网络级丢包: ${e.message}`);
    return false;
});

✅ 5. 验收标准 (Acceptance Criteria)

  1. 代码可正常编译，farm.js 与 friend.js 完全接管了新的 common.js 包装器。
  2. 挂机机器人能正常起服，未出现 `TypeError: withTimeout is not defined` 等未引用的报错。
  3. 在网速恶劣或者故意断网断连期间，测试农场种植和好友偷菜能够输出 Timeout 日志，而不是 Node 服务直接僵死卡住。
  4. 常规抢收不会因为重试时间变长而被截胡。
