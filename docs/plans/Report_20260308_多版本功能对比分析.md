## 二、🚨 最关键差异：WASM 加密协议

> **⚠️ 紧急：这是当前最高优先级的适配任务。三个参考版本均已完成，本项目未集成将导致无法正常通信。**

### 2.1 新加密协议核心原理

根据新协议版本 `功能和BUG记录.md` 的详细逆向记录：

1. **加密方式**: 从 `.tqapkg` 小游戏分包中提取 `tsdk.wasm`（119KB），含 C/C++ 编译的高强度加密算法
2. **非对称模式**: **请求体必须加密**，响应体保持明文 Protobuf
3. **WASM 导出映射**:

| 导出符号 | 函数名 | 作用 |
|----------|--------|------|
| `exports.E()` | init_runtime | 初始化 WASM 运行时 |
| `exports._` | generateTokenRaw | 签名/Token 生成 |
| `exports.J` | encryptRaw | 请求体原地加密 |
| `exports.K` | decryptRaw | 请求体原地解密 |
| `exports.z` | createBufRaw | 分配 WASM 内存 |
| `exports.A` | destroyBufRaw | 释放 WASM 内存 |
| `exports.v` | memory | WASM 内存对象 |

### 2.2 需要集成的文件

| 文件 | 大小 | 作用 |
|------|------|------|
| `crypto-wasm.js` | ~3.5KB | WASM 加载器/加密封装层 |
| `tsdk.wasm` | 119KB | 核心加密二进制模块 |

### 2.3 集成改动要点（参考 P大版本 network.js）

**改动前（本项目当前实现）**：
```javascript
function encodeMsg(serviceName, methodName, bodyBytes) {
    const msg = types.GateMessage.create({
        body: bodyBytes || Buffer.alloc(0),
        // ...
    });
    const encoded = types.GateMessage.encode(msg).finish();
    clientSeq++;
    return encoded;
}
```

**改动后（集成 WASM 加密）**：
```javascript
const cryptoWasm = require('./crypto-wasm');

async function encodeMsg(serviceName, methodName, bodyBytes) {
    let finalBody = bodyBytes || Buffer.alloc(0);
    if (finalBody.length > 0) {
        finalBody = await cryptoWasm.encryptBuffer(finalBody);
    }
    const msg = types.GateMessage.create({
        body: finalBody,
        // ...
    });
    const encoded = types.GateMessage.encode(msg).finish();
    clientSeq++;
    return encoded;
}
```

> **⚠️ 重要提示**: 响应体不需要解密！新协议版本的逆向记录明确指出，对服务端下发的明文数据调用 `decryptBuffer` 反而会破坏数据，导致 Protobuf 解码异常。

## 四、网络层核心差异详解

### 4.1 消息编码流程对比

**P大版本（已适配新协议）**：
```
bodyBytes → cryptoWasm.encryptBuffer(bodyBytes) → GateMessage.encode → ws.send
```

**本项目（旧协议，待适配）**：
```
bodyBytes → GateMessage.encode → ws.send  （无加密步骤）
```

### 4.2 消息解码流程

**四个版本一致**：服务端响应体为明文 Protobuf，直接 `GateMessage.decode(buf)` 即可。无需解密。

### 4.3 本项目独有的高级网络机制（三版本不具备）

1. **令牌桶全局限流**：334ms 基础间隔 + ±20% 随机抖动 → 防止「过匀速」被识别为机器人
2. **双队列优先级**：`urgentQueue` + `normalQueue`，紧急请求优先发送（偷菜/抢收）
3. **Anti-Starvation 防饿死**：连续处理 10 个紧急请求后强制释放 1 个普通请求
4. **断路器模式**：检测到 `errorCode === 1002003` 风控 → 全局熔断 → 所有队列清空
5. **请求积压自适应丢弃**：按方法类型设置 TTL（好友列表 3s / 常规 5s / 核心操作 15s）
6. **openId 参数支持**：WS 连接支持传入 `openId`（wx 平台）
7. **防雪崩重连抖动**：`5000 + Math.random() * 20000` ms → 5~25秒随机延迟
