# 代码审计报告 (v3.9.7~v3.9.9)

## 0. 审计综述
本次审计重点考察了近期上线的 **WebSocket 令牌桶限流 (3QPS)**、**紧急通道 (Urgent Path)**、**风控自恢复休眠** 及 **SQLite 性能增强**。总体结论：代码实现健壮性较高，防御性编程意识强，能有效应对高频操作带来的封禁风险。

---

## 1. 核心模块评估

### 🛡️ 令牌桶限流 (`network.js`)
*   **实现**: 采用 334ms 帧间隔排队，成功消除瞬时高频 API 调用。
*   **亮点**: 实现了 `normalQueue` 与 `urgentQueue` 双队列，并带 **MAX_CONSECUTIVE_URGENT (10)** 防饿死机制，设计非常成熟。
*   **潜在风险**: `cleanup()` 虽然清空了队列，但若正在进行中的 `setTimeout` 未能及时清理（极端情况），可能存在微小的竞态。
*   **优化建议**: 在 `drainQueue` 循环中增加对 `ws.readyState` 的持续监听，确保在重连间隙不产生无效等待。

### 🚨 紧急通道与 Promise 合并 (`farm.js`)
*   **实现**: `getAllLandsUrgent` 引入了 `LANDS_CACHE_TTL_MS (500)`。
*   **评估**: 这是一个关键优化。通过 `landsFetchPromise` 合并 500ms 内的所有地块查询，有效防止了「抢收时刻」多个地块同时成熟触发的并发查询风暴。
*   **建议**: 目前 `熔断清理` 逻辑仅在 catch 中设置 `landsFetchPromise = null`。建议在 Promise 成功结束后也主动通过定时器清理，确保缓存不过期（虽然现有逻辑已覆盖，但可更显式）。

### 🧊 风控自恢复休眠 (`runtime-engine.js` / `models/store.js`)
*   **实现**: 拦截错误码 `1002003`，账号进入 30 分钟休眠并持久化 `suspendUntil`。
*   **亮点**: 持久化逻辑确保了服务器重启后账号不会立即再次撞库被封，对保护账号安全意义重大。
*   **建议**: **[发现问题]** 目前 `data-provider.js` 在 `restartAccount` 时虽然补齐了 `code`，但未显式清理 worker 的旧状态。建议在 worker exit 回调中强制清理所有挂起的 Scheduler 任务。

## 2. 发现的潜在问题 (Edge Cases)

### q1. 防偷抢收的生命周期漂移
在 `antiStealHarvest` 中，虽然有 65 秒的「漂移判定」，但在高延迟环境下（3QPS 排队可能导致几秒延迟），若 `matureInSec` 刚好跨过判定边界，可能会误判为「人为干预」而取消。不过这属于「宁可漏掉不可错杀」的安全设计，符合该项目理念。

### q2. SQLite 争用优化
增加 `busy_timeout = 5000` 是正确的。但由于账号系统目前开始频繁更新 `auth_data`（如每次登录更新 code），在大规模账号并发登录时，数据库写锁仍可能成为瓶颈。

## 3. 优化建议清单 (Next Steps)

1.  **[稳定性] 显式 Worker 状态同步**: 
    在 `worker-manager.js` 收到销毁指令时，增加一个强制心跳探测，确保子进程确实已释放端口/资源后，主进程再进行变量删除。
2.  **[性能] 树状缓存结构**:
    将地块数据由单一 Promise 缓存升级为按 `accountId` 隔离的对象缓存，避免多账号间的全局变量污染（虽然目前的 worker 隔离已解决，但代码层面可更清晰）。
3.  **[安全] 智能延迟抖动**:
    目前令牌桶是 334ms 恒定速度。建议增加 ±20% 的 `random jitter`。虽然现在 3QPS 已很安全，但规律的 334ms 采样点理论上仍可被腾讯的风控行为分析识别。
