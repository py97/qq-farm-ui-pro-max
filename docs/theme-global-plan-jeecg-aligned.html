全局主题统一框架计划（JeecgBoot 策略对齐版）

目标：仅做 UI 主题体系重构，不改业务逻辑与功能行为；尽量复用现有主题逻辑，走“最小侵入、可回滚、分阶段”方案。

1. 参考基线与当前现状

JeecgBoot 主题策略（可迁移部分）

  • 中心化配置：`projectSetting + appStore` 做单一主题状态源。
  • 分层处理器：`initAppConfig` 初始化，`handler` 负责设置项事件，`theme/*` 负责 CSS 变量与暗黑适配。
  • 颜色白名单：系统主题/顶栏/菜单各自独立色板，避免任意色导致可读性问题。
  • 模式联动：暗黑模式切换时，限制/重算部分颜色，保证对比度。
  • 能力开关：灰色模式、色弱模式作为全局开关独立存在。

你当前项目（web）主题现状

  • 已有 `app` store 与 `ThemeSettingDrawer`，具备主题模式、色板、背景联动基础。
  • 存在大量页面级配色类和硬编码，语义色未统一，主题切换影响范围受限。
  • 主题 token 主要在 `style.css`，但组件侧未统一通过语义 token 消费。
  • 布局层（`DefaultLayout.vue`）已有玻璃态变量，但和页面内局部色彩体系未闭环。

2. 改造原则（严格约束）

必须满足仅 UI 主题改造，不动接口、数据流、权限、路由、业务计算逻辑。

必须满足保留现有主题行为（light/dark/auto、色板切换、背景联动、性能模式）。

必须满足优先“加层”而不是“重写”，通过兼容层让旧样式逐步迁移。

控制变更先建立全局 token 与映射，再按优先级替换高频页面，避免一次性全量重构。

3. 新版全局主题控制方案（设计稿）

3.1 分层模型（与 Jeecg 对齐）

  1. ThemeState 层：继续由 `stores/app.ts` 管理模式、主题、背景、可访问性开关。
  2. ThemeEngine 层：新增主题处理器（初始化、切换、变量落地、联动规则）。
  3. ThemeToken 层：统一定义语义 token，输出到 CSS 变量。
  4. ThemeAdapter 层：给现有 `.glass-*`、`text-primary-*`、页面常用色提供兼容映射。
  5. ThemeUI 层：设置抽屉按“系统主题/顶栏主题/菜单主题/界面设置”结构组织。

3.2 Token 结构（语义化，不绑具体组件）

Core Tokens: --t-primary-50..950 --t-neutral-0..1000 Semantic Tokens: --t-bg-page / --t-bg-elevated / --t-bg-overlay --t-text-main / --t-text-secondary / --t-text-inverse --t-border-default / --t-border-strong --t-brand / --t-brand-hover / --t-brand-active --t-success / --t-warning / --t-danger / --t-info --t-focus-ring / --t-shadow-sm/md/lg Layout Tokens: --t-header-bg / --t-header-text --t-sider-bg / --t-sider-text / --t-sider-active --t-content-bg Effect Tokens: --t-glass-bg / --t-glass-border / --t-glass-blur --t-mesh-orb-1/2/3

3.3 主题策略

  • 系统主题（5 套）：保留现有 default/cyber/elegant/ocean/sakura。
  • 顶栏主题（白名单）：对齐 Jeecg 的 Header 色板做有限集合。
  • 菜单主题（白名单）：对齐 Jeecg 的 Sidebar 色板，深色模式下限制切换策略。
  • 模式联动：`dark` 下自动计算 hover/active/边框层级并保证可读性。
  • 可访问性：灰色模式、色弱模式独立开关，挂在根节点 class。

4. 最小改动实施计划（Plan）

阶段 目标 改动范围 对业务风险 验收点
Phase 0 主题资产盘点与冻结基线 仅扫描与文档，不改代码 无 输出映射表：现有 class/变量 -> 新 token
Phase 1 建立 ThemeEngine 与 Token Registry 新增主题管理文件，复用现有 `appStore` 状态 低 light/dark/auto + colorTheme 与当前行为一致
Phase 2 加入兼容层（旧类到新 token） `style.css`/布局样式映射，不改业务组件逻辑 低 页面视觉基本不变，主题切换覆盖范围扩大
Phase 3 设置面板重组为 Jeecg 风格策略面板 仅 UI 配置项与展示结构，保留原动作接口 中低 包含：系统主题/顶栏主题/菜单主题/界面设置
Phase 4 高频页面渐进替换硬编码色彩 优先 Settings/Accounts/Dashboard/Friends 中 替换为语义 token，UI 无功能回归
Phase 5 回归与对比度校验 视觉回归脚本 + 手工场景清单 可控 主题切换、暗黑/浅色、灰色/色弱全部通过

5. 关键设计决策（与你当前代码对齐）

  1. 不替换 `stores/app.ts` 的现有数据结构，只补充“主题处理器函数”并在 watch 链接入。
  2. 不改变现有 API（`/api/settings/theme`）字段协议，新增字段采用可选方式向下兼容。
  3. `style.css` 继续作为全局主题变量承载层，但升级为“Core + Semantic + Layout + Effect”结构。
  4. 对现有 `.glass-panel/.glass-text-*` 保持兼容，先接 token，不强制页面立即改 class。
  5. 设置抽屉保留现有交互入口和持久化，只重排信息架构与视觉层级。

6. 风险与回滚

  • 风险页面局部硬编码颜色过多，部分状态色在主题切换下可能对比度不足。
  • 风险图表/第三方组件可能仍使用固定色，需要单独 token 接入。
  • 回滚按 Phase 提交，任一阶段可独立回退，不影响业务逻辑。
